## 用户中台

> 关键技术点：分库分表、缓存一致性、rpc、分布式 id

用户中台的意义在于为自家的多个 app 提供用户服务，打通同系应用的用户数据。

- 统一管理用户数据。

- 要满足 app、web、小程序的多端接入。

- 满足高性能、高并发，高可用。

其性能瓶颈多在于 mysql 的，所以不可避免的要读写分离、分库分表，同时增加分布式缓存，那这就必须要解决缓存一致性。

同时要考虑 redis 单机内存不足的情况，考虑使用集群分片存储，[内存预估工具](http://redis.cn/redis_memory/)

细化到接口层面，主要有以下业务场景：

- 单用户查询，不建议使用本地缓存
- 批量用户查询，分库键的策略
- 用户更新，缓存和数据库的一致性
- 用户注册，分布式唯一 id 的生成

```java

 * 命名请以服务名去掉前缀 live 为规范

public class UserProviderCacheKeyBuilder extends RedisKeyBuilder{
    private static final String USER_INFO_KEY = "userInfo";
```

根据上面的类命名规范，通过比对类名是否包含应用名来判断是否需要加载该 key 生成器。

```java
@Slf4j
public class RedisKeyBuilderLoadCondition implements Condition {

    private static final String PREFIX = "live";

    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        String appName = context.getEnvironment().getProperty("spring.application.name");
        if (StrUtil.isEmpty(appName)) {
            log.error("Unable to get the application name, so the match does not take the key builder.");
            return false;
        }
        try {
            Field field = metadata.getClass().getDeclaredField("className");
            field.setAccessible(true);
            String keyBuilderName = (String) field.get(metadata);
            List<String> keyBuilderNameSplitList = Arrays.asList(keyBuilderName.split("\\."));
            String classSimplyName = PREFIX + keyBuilderNameSplitList.get(keyBuilderNameSplitList.size() - 1).toLowerCase();
            boolean matchResult = classSimplyName.contains(appName.replaceAll("-", ""));
            log.info("keyBuilderName is [{}],matchResult is [{}]", keyBuilderName, matchResult);
            return matchResult;
        } catch (Exception e) {
            throw new RuntimeException();
        }
    }
}
```

#### 数据一致性问题

写时修改缓存，无论先写哪个，都会出现数据不一致的情况，因为两个写操作不是原子的，只能通过加锁来保证一致性，而这又影响性能。

所以考虑将修改缓存该卫生删除缓存，等到读的时候再将缓存更新。

那到底是先删除缓存还是先改库呢？

先改库的话，如果再缓存删除之前有读请求，就会出现数据不一致的现象。

先删缓存，再改库，如果再改库前有读操作，那么旧的数据就会再生成一个缓存，造成数据不一样，同样的如果是主从数据库读写分离的架构，主从同步延迟依然会导致读操作读到旧数据，从而缓存是旧的。

由此，**延迟双删**诞生，其特点是只保证最终一致性，基于 Canal 的订阅 binlog 的方式可以实现强一致性。

- 添加标签：或运算

- 删除标签：先对标签取反再做与运算

- 是否有某个标签：与运算之后 = 标签

- 没有某个标签：与标签或运算 与  0

### 发送验证码

发送验证后 60s 内禁止重复发送，使用 redis 存储验证码，既能禁止用户重新发送（类似分布式锁的思想），又方便校验验证码是否正确。

生成验证码使用随机数来实现，如生成 6 位数的随机验证码。

实际发送的操作要借用第三方短信云平台。

发送成功后要记录到mysql 持久化。

## 问题汇总

### Netty

#### RefCnt=0 问题

[参考资料](https://emacsist.github.io/2018/04/28/netty%E4%B8%AD%E5%B8%B8%E8%A7%81%E7%9A%84illegalreferencecountexception%E5%BC%82%E5%B8%B8%E5%8E%9F%E5%9B%A0%E5%8F%8A%E8%A7%A3%E5%86%B3/)

```
io.netty.util.IllegalReferenceCountException: refCnt: 0, increment: 1
```

访问到了引用次数为 0 的对象，故障原因是编码器编写错误，多加了`ctx.writeAndFlush(msg);`，该方法会将对象的引用次数减一，而发送时又会调用一次该方法，导致对象的引用次数为 0 。

理解编解码器的作用只是将 `byteBuf`转为我们自定义的消息体，不承担任何发送的任务。

#### ChannelHandler.Sharable注解

`@ChannelHandler.Sharable` 用于标识一个 ChannelHandler 是否可以在多个 ChannelPipeline 中共享使用。
这意味着可以将一个实例添加到多个 ChannelPipeline 中，而不会出现线程安全问题。

### RocketMQ

#### 从何处开始消费

在 Apache RocketMQ 中，`ConsumeFromWhere` 是用来指定消费者从何处开始消费消息的枚举类型。它包括以下几个枚举值：

- `CONSUME_FROM_FIRST_OFFSET`：从队列的最开始位置开始消费。
- `CONSUME_FROM_LAST_OFFSET`：从队列的最末尾开始消费，即跳过已经消费过的消息。
- `CONSUME_FROM_TIMESTAMP`：从指定时间戳开始消费。

### MySql

#### mysql 主从架构的问题有哪些？

最大的问题就是**主从延迟**：

- 保证主从节点的机器配置相同，高带宽，3-5 个从节点。

- 分片导致很多批量的命令无法执行。

- 节点过多时信息同步会比较慢，采用流言算法。
