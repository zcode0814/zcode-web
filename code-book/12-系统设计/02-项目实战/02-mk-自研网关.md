# 基础理论篇

## 网关基本概念

所有后台服务的**大门**，处理 `HTTP` 和 `WebSocket` 两种 API 请求。

### 优缺点

网关大大简化了客户端的操作，所有请求只要发送到网关即可，不需要考虑具体的后端服务。

而后端开发人员只需要关注业务，不需要考虑负载均衡等问题。

但是其**中心化架构**也成为微服务的瓶颈，毕竟网关一崩全盘皆凉。

这对网关的性能要求也就上了一个台阶，如果不是异步或同步非阻塞处理客户端的请求，那性能堪忧。

### 核心功能

- 路由转发
- 协议转换
- 负载均衡
- 黑白名单
- 拦截过滤（鉴权）
- 监控统计
- 熔断限流
- 日志
- 优雅下线

### 对比选型

#### Nginx + OpenResty

同时作为**流量网关**和**业务网关**，性能非常好。

但是其使用 Lua 语言扩展功能，学习成本高。

#### Kong

基于OpenResty，有管理界面。

支持 grpc、http、dubbo 协议。

#### Spring Netflix Zuul

**1.0 版本**

中小厂落地案例丰富，基于同步阻塞 IO，性能差

**2.0版本**

性能提升，但是进入维护期，前景不明。

#### Spring Cloud Gateway

异步非阻塞 IO，性能非常好。

不直接支持 Dubbo 协议，即不能直接路由到 dubbo 服务，需要 web 层支持。

## 为什么要自研

现有开源框架的不足之处：

- 现有组件的附加功能太多

- 技术栈（开发语言）不符合团队

- 性能参差不齐

自研的优势：

- 因需制宜
- 研发需求及进度可控
- 运维成本低

### 功能点详解

1. 接受外部请求并做合法性校验
2. 在请求发送和返回的过程中执行各种插件，如黑白名单过滤。
3. 多个插件连在一起成为一条规则。
4. 一个规则可以配置多个路径。

### 技术栈
