- condition 的底层实现

在Java中，`Condition`是与`Lock`配合使用的线程等待/通知机制，它提供了类似`Object.wait()`和`Object.notify()`的功能。`Condition`的底层实现通常是基于等待队列（Wait Queue）和通知机制来实现线程的等待和唤醒。

具体来说，`Condition`的底层实现通常包括以下几个关键组件：

1. **等待队列**：每个`Condition`对象都会维护一个等待队列，用于存放调用`await()`方法被阻塞的线程。

2. **等待和唤醒操作**：当线程调用`await()`方法时，会释放锁并进入等待状态，同时将线程加入到等待队列中。当其他线程调用`signal()`或`signalAll()`方法时，会从等待队列中选择一个或全部线程唤醒，并尝试重新获取锁。

3. **与Lock的配合**：`Condition`通常与`Lock`接口的实现类（如`ReentrantLock`）一起使用，通过`Lock`的`newCondition()`方法创建`Condition`对象。

总体来说，`Condition`的底层实现是基于等待队列和线程通知机制来实现线程的等待和唤醒，从而实现线程间的协作和同步。

- spring 注册为 bean

**自定义Bean注册器**：实现`BeanDefinitionRegistryPostProcessor`接口，通过编程方式注册Bean。

```java
@Configuration
public class CustomBeanRegistry implements BeanDefinitionRegistryPostProcessor {

    @Override
    public void postProcessBeanDefinitionRegistry(BeanDefinitionRegistry registry) {
        // 创建一个新的Bean定义
        GenericBeanDefinition beanDefinition = new GenericBeanDefinition();
        beanDefinition.setBeanClass(MyCustomBean.class);
        
        // 注册Bean定义
        registry.registerBeanDefinition("myCustomBean", beanDefinition);
    }

    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) {
        // 可以在这里进行其他的BeanFactory后处理操作
    }
}
```
